# release — リリース手順とゲート

リリース = `mise run release` によるタグ作成と push。タグは打ち直しのコストが高く、
push は origin への公開を意味する。以下の順序を厳守する。

## 手順

1. **全件検証（省略不可）**: `pongo run`（**統合テスト 10-integration_spec を含む全件**）と
   `pongo lint` を実行し、すべて green であることを確認する
2. **バージョン更新**: 3 箇所（rockspec ファイル名 / `package_version` / handler.lua の
   `VERSION`）を同じ値に更新してコミットする。詳細は `docs/05-release.md` の表を参照
3. **リリース実行**: `mise run release`（整合性チェック → `v<version>` タグ → origin へ push）。
   **origin にローカルの main 全体が公開される**ことをユーザーが把握しているか確認する
4. **リリース後確認**: GitHub Actions の Release ワークフローが成功し、Releases ページに
   `v<version>` が作成されたことを確認して報告する

## ステップ 1 を飛ばす合理化への反論

| 言い訳 | 現実 |
|---|---|
| 「バージョン文字列だけの変更だからロジックに影響しない」 | 変更内容ではなく**リリースというイベント**がゲートの根拠。タグ済みコミットが未検証なら、その保証は永久に欠ける |
| 「時間がない・急ぎと言われた」 | 統合テスト込みでも数分。壊れたタグの取り消し・再リリースの方がはるかに遅い |
| 「単体テストは通ったので統合は省略」 | このプラグインの実バグは統合テストだけが検出した実績がある（キャッシュ・IdP 連携はモック単体では再現しない） |
| 「CI が後で検証してくれる」 | Release ワークフローの検証は luarocks make のみ。テストは走らない |

## Red Flags — 出たら手を止める

- テストを流さずに `git tag` / `mise run release` を打とうとしている
- 「後で検証する」と言いながらタグを push しようとしている
- バージョン 3 箇所のどれかを手動確認せずに済ませようとしている（`mise run release` が
  検知するが、失敗してから直すより先に揃える）
